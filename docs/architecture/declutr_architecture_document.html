Declutr — AI-Powered Intelligent Digital Life Vault

Principal Engineering Architecture Reference

Version 2.0 Classification: Technical Architecture Intellectual Memory Core Adaptive Privacy Enabled

1. System Overview

Declutr is an AI-powered digital life vault engineered to help users securely store, organize, connect, and retrieve the massive, fragmented volume of information they accumulate throughout their digital lives.

Traditional digital storage solutions rely on folder paths, arbitrary filenames, file types, and manually managed tags to retrieve information. However, human memory is contextual, associative, and narrative-driven. Users recall items not by their exact paths, but by their real-world contexts and relationships (e.g., "that hotel booking receipt I saved before my Mumbai trip" or "the document related to my summer internship").

Declutr shifts the digital storage paradigm. Rather than acting as a simple encrypted database of files, it serves as an Intelligent Personal Digital Memory System. It answers key semantic questions about every piece of stored information:

  • WHAT the item contains and what it represents.
  • WHY the item matters to the user (its long-term value and utility).
  • WHAT broader context or project the item is connected to.
  • WHAT other files, events, or entities the item relates to.
  • HOW the user interacts with this information over time.
  • HOW the user is likely to remember and search for information.
  • WHETHER the current session or access behavior suggests a security anomaly.
ℹ
Core Privacy Mandate: Security and zero-knowledge encryption form the enabling foundations of the platform, protecting user privacy at all times. However, security is not the sole product identity. Declutr is defined by its ability to intelligently understand and connect the shards of a user's digital existence.

Product PillarsDesign Ideology

2. Core Product Principles

Declutr's architecture is structured around five core intelligence pillars and a sixth supporting behavioral security layer. Privacy and encryption underpin the entire system.

🧠
Content Intelligence
Extract structured semantic features, OCR text, document layout concepts, visual objects, and audio segments. Automatically categorize and summarize content upon ingestion.
🎯
Intent Intelligence
Determine the utility of an item (e.g., reference material, active planning document, legal archive, transaction record) to predict when, why, and how a user will retrieve it.
🔗
Relationship Intelligence
Dynamically associate independent digital artifacts. Detect temporal proximity, entity matching, and contextual links (e.g., linking a boarding pass, hotel receipt, and a screenshot of restaurant recommendations into a unified "Trip" context).
👤
Persona Intelligence
Compile behavior signals, search choices, and content patterns into a private, probabilistic "Reverse Persona" that guides contextual ranking, relevance tuning, and intent inference.
🔍
Retrieval Intelligence
Synthesize traditional keyword search, metadata filters, semantic vectors, and relationship graphs to support natural-language query resolution and temporal reasoning.
🛡
Behavioral Security
Construct passive behavioral baselines (session properties, access volume, download rates, page navigation) to evaluate risk dynamically and trigger adaptive authentication challenges without disrupting the user flow.

Authentication & User Identity

Zero-Knowledge AuthPasskeys

6. Authentication & User Identity

Declutr implements a zero-knowledge authentication scheme using **Secure Remote Password (SRP-6a)** as the primary path, supplemented by **WebAuthn / Passkeys** as modern alternatives. This ensures that credential validation occurs without transmitting passwords to the server, protecting the system against credential leaks.

SRP-6a Authentication Exchange

1
Client Initial Request
The client submits the user's email hash and an ephemeral public value A = g^a mod N.
2
Server Challenge Response
The server matches the email hash to retrieve the SRP salt and verifier v. It generates ephemeral b and returns the salt alongside public value B = (kv + g^b) mod N.
3
Shared Key Calculation
Both client and server calculate a shared cryptographic proof key S using client secret a and server secret b.
4
Proof Verification & Token Issue
The client submits proof M1. The server validates M1, responds with proof M2, and issues a short-lived access JWT (15 mins) and an httpOnly refresh token. The client uses the derived master key to decrypt its local vault keys.

Vault & Digital Item Model

Vault DomainsVersioning

7. Vault & Digital Item Model

A vault represents an isolated cryptographic workspace. Within a vault, a **Digital Item** is the atomic object of management. Rather than mapping strictly to a simple file, a Digital Item abstracts a logical piece of digital memory, containing versions, extracted content, intents, contexts, and relationships.

🔒
Logical Isolation
Each vault possesses an independent 256-bit Vault Key (VK). This key is never accessible to the backend; it is stored as ciphertext wrapped by the Master Vault Key (MVK).
⏳
Immutable Version History
Changes to a Digital Item create a new version. Versions reference distinct S3 storage paths, preventing modification of historical records. Active pointers resolve the current version.

File Upload & Content Ingestion

Multipart UploadS3 Pre-signed

8. File Upload & Content Ingestion

Declutr uploads files securely using client-side encryption and S3 pre-signed URLs, keeping transfer loads away from backend processing servers.

1
Generate Keys & Encrypt Client-Side
The client generates a unique 256-bit File Key (FK) and encrypts the file locally with AES-256-GCM. It divides files larger than 4MB into chunks.
2
Acquire Pre-signed Direct Upload Links
The client requests S3 pre-signed upload URLs from the backend for each chunk. The backend registers the upload job with an UPLOADED state.
3
Direct Cloud Storage Transfer
The client uploads encrypted blocks directly to S3-compatible storage. If a network interruption occurs, the client queries chunk status to resume from the last successful part.
4
Commit & Enqueue AI Ingestion
The client submits the commit metadata. The backend verifies block sizes, updates the record state to QUEUED, and triggers the asynchronous Content Intelligence pipeline via Redis/BullMQ.

Metadata Architecture

Sealed MetadataData Categorization

10. Metadata Architecture

Declutr divides data into structured layers, separating user content from operational metadata to enforce zero-knowledge isolation where required.

👤
User-Provided Data
Plaintext files, filenames, user tags, and notes. This data is encrypted client-side using the Vault Key before transmission, remaining completely opaque to the backend database.
🤖
AI-Generated Data
Derived tags, entity fields, intent tags, relationship logs, and semantic embeddings. In Private Mode, these are encrypted client-side. In Enhanced AI Mode, these are processed by secure backend workers and stored as encrypted database records.
🖥
System-Generated Data
Identifiers (UUIDs), storage block paths, ciphertext file sizes, upload timestamps, and active version pointers. This data is visible in plaintext to the server to support query routing and storage quota management.

AI Organization Engine

Automated StructuringInference Loop

11. AI Organization Engine

The AI Organization Engine organizes the vault automatically. It groups files into logical folders and assigns smart tags without requiring manual filing from the user.

This engine acts as the orchestrator for the Content Intelligence Pipeline. It receives the raw text and images extracted during ingestion, processes them through the classification models, and generates suggested tags, topic categories, and summary fields. These suggestions are presented to the user for confirmation, establishing a system-wide feedback loop.

Intent-Aware Organization Engine

Intent ClassificationContextual Utility

12. Intent-Aware Organization Engine

The Intent-Aware Engine distinguishes between the physical classification of a file and its practical utility to the user.

Defining Core Intent Dimensions

Concept Definition Marriott Screenshot Example
Category Physical classification of the item's content. Image / Screenshot of Travel Reservation
Intent The inferred purpose of saving this item. Booking Reference, Travel Planning, Expense Claim
Context The real-world situation or project. Mumbai Business Trip - August 2026
Relationship Direct links to other files and events. Flight tickets, Payment receipt, Calendar event

Handling Intent Uncertainty

Intent inference is probabilistic rather than absolute. The engine generates multiple potential intents with associated confidence scores. Suggestions remain in an unreviewed state until the user accepts, corrects, or rejects them. The feedback loop updates the Reverse Persona model to refine future classification accuracy.

Relationship & Context Engine

Relational GraphsContextual Links

13. Relationship & Context Engine

The Relationship Engine links separate digital items using their shared context. For example, a flight ticket, a hotel booking confirmation, a taxi receipt, and a screenshot of a restaurant menu are grouped into a single logical context, such as a **"Mumbai Trip."**

Direct Relationship Formats

  • RELATED_TO: General association between items.
  • PART_OF: Direct hierarchy (e.g., an invoice file is part of an expense claim context).
  • MENTIONS: Shared entities (e.g., both items reference the same client or contract ID).
  • SAME_EVENT / SAME_LOCATION: Co-located in time or geography.
  • PRECEDES / FOLLOWS: Sequential workflow ordering (e.g., booking confirmation precedes boarding pass).

For the MVP, relationships are stored as relational structures in PostgreSQL. Graph databases (such as Neo4j) are excluded from the MVP scope to reduce system complexity, but remain a post-MVP scaling option.

Reverse Persona Architecture

Probabilistic ModelingContextual Personalization

14. Reverse Persona Architecture

The **Reverse Persona** is a defining intelligence layer in Declutr. It models user behavior and preferences to customize search rankings, intent classifications, and context grouping. It operates probabilistically; single actions do not lock user attributes into rigid states.

Behavioral Signals

  • Content types saved most frequently.
  • Recently searched terms and selected results.
  • Tag corrections and rejected AI suggestions.
  • Temporal interaction patterns (e.g., files accessed during work hours vs. weekends).
[User Activity Event] -> [Signal Collector] -> [Weight & Decay Evaluator] │ ▼ [Personalization Context] <- [Persona Attributes] <- [Evidence Store]

Attribute Decay & Privacy Controls

Attributes update dynamically using recency decay curves. If a user stops saving software development screenshots, the interest weight decays over time. The user maintains control through explicit settings, including options to clear specific persona attributes, opt out of personalization, or purge all behavioral history.

Behavioral Authentication & Adaptive Risk Engine

Adaptive SecurityRisk Evaluation

15. Behavioral Authentication & Adaptive Risk Engine

The Behavioral Authentication Engine estimates session risk using passive behavioral signals. This system is **separate from the Reverse Persona Engine**; they use different data stores and processing pipelines to isolate personalization from security controls.

Behavioral Risk Architecture

1
Passive Signal Collection
Tracks session attributes (IP subnet, client device fingerprint), navigation speed, data request rates, and sensitive action patterns.
2
Risk Scoring
Combines incoming signals to generate a real-time session risk score from 0.00 to 1.00.
3
Adaptive Challenge Selection
Executes response actions based on the active risk score. Standard password/passkey verification remains the authoritative source of identity.

Adaptive Security Actions

Risk Level Score Range Security Action
LOW RISK 0.00 - 0.30 Allow all operations; maintain background monitoring.
MEDIUM RISK 0.31 - 0.60 Prompt for re-authentication for sensitive actions (e.g., key exports).
HIGH RISK 0.61 - 0.85 Require WebAuthn / Passkey verify or MFA check to proceed.
CRITICAL RISK 0.86 - 1.00 Block active operation, terminate current session, and log security alert.

Search & Natural-Language Retrieval

Hybrid Searchpgvector

16. Search & Natural-Language Retrieval

Declutr resolves natural-language queries by combining keyword search, metadata filtering, semantic vectors, and relationship mapping.

[User Query: "Marriott screenshot before my Mumbai trip"] │ ▼ [Query Understanding & Parse (Type=screenshot, Entity=Marriott, Context=Mumbai Trip)] │ ▼ [Search Execution: Keyword Match + Vector Embeddings + Metadata Filter] │ ▼ [Graph Expansion: Retrieve items linked to "Mumbai Trip"] │ ▼ [Persona Ranking: Sort by user access history and preferences] │ ▼ [Result Fusion: Display verified relevant records]

In **Enhanced AI Mode**, semantic search queries are parsed on the backend database using **pgvector**. In **Private Mode**, search queries are processed locally on the client device using WebAssembly-compiled embedding runtimes, querying local caches without sending query text to the server.

Privacy & Encryption Architecture

Privacy ModesZero-Knowledge

17. Privacy & Encryption Architecture

To support advanced AI capabilities alongside zero-knowledge data protection, Declutr allows users to select between two distinct privacy configurations.

🛡
Private Mode (Default)
  • All files are encrypted client-side using unique AES-GCM keys before upload.
  • AI inference (OCR, classification, tag suggestion) runs on the client device using local ONNX runtimes.
  • The server stores only encrypted payloads, ciphertext metadata, and blind tokens for keyword matching.
  • No plaintext data or unencrypted metadata is ever transmitted to the server.
🚀
Enhanced AI Mode (Explicit Opt-In)
  • Allows selected files to be decrypted temporarily within secure backend sandboxes for advanced AI processing.
  • Enables deeper intent classification, relationship discovery, and server-side semantic search.
  • Requires explicit user consent on a per-folder or per-vault basis.
  • Users can revoke access at any time, which deletes derived plaintext metadata from server caches.

Background Jobs & AI Processing

BullMQAsync Pipeline

20. Background Jobs & AI Processing

AI ingestion tasks run asynchronously to ensure they do not block core vault operations. The upload module writes files to S3 and commits metadata records in an UPLOADED state, then enqueues processing jobs in **Redis/BullMQ**.

AI processing workers pick up queued jobs to run text extraction, OCR, classification, and vector embedding generation. If an AI job fails or runs slowly, the user can still download, move, or delete the file. AI metadata enrichment is non-blocking and fails gracefully.

Security Model

Zero-TrustData Isolation

21. Security Model

Declutr operates on zero-trust principles. All database queries use PostgreSQL **Row-Level Security (RLS)**, restricting database access to the logged-in user context. Service access tokens are short-lived, scopes are minimized, and data is isolated cryptographically across vaults.

Threat Model

Threat VectorsSTRIDE Mitigations

22. Threat Model

Expanding Declutr to support AI organization and vector searches introduces new threat vectors that must be mitigated.

Malicious File Uploads & InjectionCritical Risk
Threat: Attackers upload files designed to trigger parser exploits in downstream OCR or text-extraction libraries, or prompt injections in LLM pipelines.
Mitigation: Parse files in isolated backend sandboxes (Docker/WASM runtimes) with strict memory limits and no network access. Sanitize extracted text before processing.
AI Metadata & Embedding LeakageHigh Risk
Threat: Compromise of the database exposes semantic vectors and AI tags, leaking sensitive user info even if raw files remain encrypted.
Mitigation: In Private Mode, store vector embeddings and AI metadata as client-encrypted blocks. In Enhanced AI Mode, encrypt embedding indices and restrict access via RLS.
Behavioral Auth False PositivesMedium Risk
Threat: Legitimate user actions (e.g., bulk exports during a trip) trigger high risk scores, blocking access or causing session lockouts.
Mitigation: Implement a progressive grace period. Prompt users with WebAuthn/Passkey challenges rather than executing outright blocks, unless risk scores reach critical levels.

Observability & Error Handling

State TrackingSystem Health

23. Observability & Error Handling

Declutr monitors ingestion jobs by tracking progress states and logging errors at each step of the pipeline.

Ingestion Pipeline Job States

  • UPLOADED: File blocks successfully transferred to S3; metadata registered.
  • QUEUED: Processing task registered in BullMQ, waiting for an available worker.
  • PROCESSING: Worker actively running OCR, classifications, and vector embeddings.
  • PARTIALLY_PROCESSED: Core text extracted, but downstream operations (e.g., transcription) failed.
  • COMPLETED: Metadata enriched, vectors generated, and search index updated.
  • FAILED: Unrecoverable parsing error; fallback to client-side indexing.
  • RETRYING: Transient error encountered; job scheduled for retry with exponential backoff.

26. Scalability & Evolution Plan

Scale-Out Roadmap

1

Monitor CPU, memory, and queue lengths in production to locate high-load operations (typically OCR, transcription, and vector searches).
